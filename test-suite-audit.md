### Key facts
- bun 1.4.0 default `bun test`: ONE process, test files sequential, preload runs once per PROCESS (verified via PID/counter files)
- `--max-concurrency` (default 20) limits concurrent TESTS within the runner, not processes; CI's `--max-concurrency 1` keeps the single-process model, so the preload runs exactly once in CI
- `--isolate` re-evaluates the preload once per FILE in the same process (~93ms/file warm); `--parallel=N` spawns N worker processes (implies --isolate), each paying one cold preload (~0.52s) then warm per-file re-evals
- Preload cost on this machine: ~515-575ms per cold process (repo+preload 0.610-0.672s vs repo no-preload 0.095-0.098s vs temp no-preload 0.085-0.092s on a trivial 1-test file)
- Attribution: jsdom import = 449.5ms (~85-90% of total); radix-ui umbrella (55 @radix-ui packages) = 65-130ms; react/react-dom family = ~40ms; react-native mocks = ~4ms; new JSDOM() = 24ms
- CI amortization: 0.52s / 658 files = ~0.8ms per file — negligible; the preload is NOT a CI-suite bottleneck under the current lane
- The real cost is the dev inner loop: every single-file `bun test foo.test.ts` pays +0.52s, and 0 of the 427 root test/ files use react/jsdom/radix — the preload only serves the 231 apps/desktop files
- apps/mobile has 0 test files, so the mobile react-native mock block in the preload currently protects nothing
- `bun test --config=<file>` (equals form required) cleanly swaps bunfig per lane; `--config <file>` with a space is misparsed as a test filter and runs zero tests
- Hypothetical --isolate full suite: ~+61s; hypothetical --parallel with 16 workers: ~4.3s total preload wall, amortized across workers
